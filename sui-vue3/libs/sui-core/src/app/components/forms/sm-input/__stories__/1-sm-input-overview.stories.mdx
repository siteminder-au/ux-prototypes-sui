## Usage
Users can enter and interact with data via input fields. Form layout is frequently utilized, but it can exist outside of a standard form.

  ### When to use
  - Use a text input field when a single line of text, rather than a paragraph, is the desired user input.
  - Use a text area field when more than one sentence is anticipated from the user.
  - Use mixed-input fields if additional components, such as dropdown menus and pickers, can be nested in text fields to facilitate input.

## Formatting

### Anatomy

  **1: Container** - Region of interactivity for input.

  **2: Label text** - Label text is used to tell users about the information that is required for a text field. All input fields should be labeled.

  **3: Helper icon (optional)** - Provides more context on the input label via tooltip/popover.

  **4: Input text** - Information entered into the text field.

  **5: Leading icon/Prefix icon (optional)** - Describe the type of input required by a text field.

  **6: Trailing icon/Suffix icon (optional)** - Can be additional control over typed text, such as clear, hide/show, and so on.

  **7: Helper text** - Provides more context or validation on the input label.

  <img src={anatomy} className="img-padding" alt="Anatomy of an input" width="420" />

### Sizing
  For desktop, input height is **40px**, while for mobiles, input height is **48px**.

  The touch area of the form field must be a minimum of 48px high to meet AA accessibility requirements (e.g. touch-enabled laptops). There is no default width size for the input. **It is very important that you indicate what the ideal width is in your designs and communicate this clearly to engineering.**

  <img src={sizing} className="img-padding" alt="Sizing for desktop and mobile" />


## Layout
### Labels
  Form labeling that is effective assists users in understanding what information to type into a text field.

  **Best practices for accessibility on form labels**
  1. When an input receives focus, labels must receive focus as well.
  2. On focus, labels must be announced by the screen reader.

<img src={layoutLabels} className="img-padding" alt="Sample for input label on SUI primary blue color." width="420" />

### Icon signifier
  Icons can describe the type of input required by a text field.
  Touch targets for nested components can also be represented by icons. A calendar icon, for example, can be pressed to reveal a date picker.

  <img src={layoutIconSignifier} className="img-padding" alt="Sample of calendar icon signifier describing visually the type of input required by a text field." width="420" />

### Input text
  1. **Input text** - Text entered by the user into a text field. <br />
  2. **Active indicator** - The position of a cursor in a field indicates the current place for text input.

<img src={layoutInputText} className="img-padding" alt="Input field on active state where there is an active indicator." width="520" />

### Required text (indicator)
  Display an asterisk (*) next to the label text to indicate that a field is necessary. Asterisk for required field default color is **$app-warning**.

  <img src={layoutRequireIndicator} className="img-padding" alt="Input field with asterisk on the input label to indicate the field is required." width="420" />

### Placeholder text
  Placeholder text serves as a hint or example of what information to enter. Placeholder text is removed as the user begins entering data into the input and should not include critical information. Use sentence-case, and write the content as concise as possible without punctuations in most cases.

  Placeholder default color is **$grey-neu-mid**

  <img src={layoutPlaceHolder} className="img-padding" alt="Input field with placeholder text." width="420" />


### Assistive elements

  #### 1. Helper Text
  Helper text is relevant information that aids the user in filling up a field. When the input is focused, helper text is placed beneath the input field container, left-aligned and can be persistent.

  It should just take up one line and be optionally displayed all the time or when in focus. Use sentence-case.
  <img src={assistiveHelperText} className="img-padding" alt="Input field with helper text placed beneath the input field container, left-aligned and persistent" />

  #### 2. Helper Icon  (optional)
  Provides more context on the input label via tooltip/popover.
  <img src={assistiveIcon} className="img-padding" alt="Input field with helper icon placed beside the field label with tooltip or popover" />

  #### 3. Error message
  An error message can provide instructions on how to resolve user errors. Error messages are displayed under the input field, replacing the helper text until the problem is resolved.
  <img src={assistiveError} className="img-padding" alt="Input field with error message under input field" width="420" />

  #### 4. Icons
  Icons can be paired with error messages to provide multiple alerts. This is useful for colorblind users as well.
  <img src={assistiveErrorIcon} className="img-padding" alt="Error icon supplementing errors placed within the right side of the input field." width="420" />

  #### 5. Character counter
  If there is a character or word limit, character or word counters should be applied. They show the character amount as well as the total character limit. This is used for the text area fields.
  <img src={assistiveCharacterCounter} className="img-padding" alt="Text field with character counter on the top right side of the field." width="420" />

  #### 6. Action
  Text area with action will have access to text link action styles. Please follow the text link states, not the text button states.
  <img src={assistiveAction} className="img-padding" alt="Text link as assistive element on the top right side of the field." width="720" />


## Placement
### Single-column
  **Recommended** - The eyes naturally go from top to bottom when adopting a one-column design. One-column forms are easier to scan and fit well on mobile devices. Multiple columns might put a cognitive load on a user, causing them to skip information inadvertently.
  <img src={placementSingle} className="img-padding" alt="One column form in wizard" />

### Semantic groups
  **Recommended** - In cases where the form has a lot of fields, try to break them into logical groups by putting some extra spaces or section titles between them. This method gives the impression of a progressive response to the user.
  <img src={placementSemanticGroups} className="img-padding" alt="Input fields broken down into into logical groups by spacings" width="520" />

### Inline fields
  **Inline fields should be used for fields that logically go together**

  **Recommended** - Stacking related data (i.e. first name, last name, addresses, and payment information creates extra friction and can disrupt a user's flow.
  <img src={placementInline} className="img-padding" alt="Address line are logically grouped together as sample for inline fields" />

The input fields will stay consistent across desktop and tablet breakpoints. On mobile breakpoints, inputs are stacked and full width of the container.

## Variants

### Input field
  Text fields should be visible and show that users may enter data. Form layout is frequently used. However, it can exist outside of a defined form.

  Available types in:

  <img src={variantInput} className="img-padding" alt="Input field variants" width="580" />

### Text area
  [Text area](https://sui-dev-v3-sui-docs.dev.siteminderlabs.com/?path=/story/components-form-input--text-and-textarea-input) is used when more than one sentence or a paragraph is anticipated by the user.

  Overflow content is wrapped into a new line in text areas. When the cursor hits the bottom of the field, they scroll vertically.

  The **minimum height set** to the text field is **122px**. There is no default width size for the text area. **It is very important that you indicate what the ideal width is in your designs and communicate this clearly to engineering.** <br /><br />

  #### Types
  1. **Fixed** - A text area with fixed width and height.
  2. **Adjustable** - A text area that may be resized vertically by the user. The adjustable text area has a resize handle icon placed at the bottom right corner, as an indicator that the field is adjustable
  <img src={variantTextArea} className="img-padding" alt="Text area types" width="720" />

### Mixed inputs
  [Mixed inputs](https://sui-dev-v3-sui-docs.dev.siteminderlabs.com/?path=/story/components-form-input--mixed-inputs) is a component variant to make input simpler, components like button, dropdown menus and pickers can be nested in text fields.

  #### Types
  1. **Interactive** - Initiates actions from the nested components (i.e., dropdown, picker, icon button, buttons)
  2. **Non-interactive** - Functions as assistive content, nested in text fields to make input easier.
  <img src={variantMixedInputs} className="img-padding" alt="Mixed input types" width="720" />

## Behaviours

  ### Scaling
  Apply adjustable container size to text fields when layouts adapt to larger screens and form factors.

  As layouts scale, minimum and maximum values for margins and paddings are set, and container dimensions so that typography adapts for easier readability.

  ### Text Overflow
  **On active state:** If the input text is too long for the width of the input container, when the cursor hits the edge of the field, it scrolls horizontally.
  <img src={behaviorOverflow} alt="text overflow behavior" width="420" />

  **On filled state:** If the input text is too long for the width of the input container, the text is truncated with an ellipsis.
  <img src={behaviorOverflowFilled} alt="Filled input state with overflow" width="420" />

  ### Validation and error messages
  If inputs are with suffix icons, the icons will change based on the message type while validating the text fields in real-time.

  When the input being entered does not meet the criteria, the helper text becomes an error message. The focus color switches to blue when a user chooses the input field and begins entering or modifying information.

  When the conditions are met, the error and warning messages are removed.

  View [Validation and debounce](https://sui-dev-v3-sui-docs.dev.siteminderlabs.com/?path=/story/components-form-input--validation-mode-and-debounce)
  <img src={behaviorErrors} alt="Input field error validation behavior" width="480" />

## Internationalisation
The arrangement of the text field is mirrored for RTL (right-to-left) languages. The label is right-aligned, with left-aligned embellishments (character count, validation marker, and error icon). Keep in mind that certain text (for example, email addresses) is not translated.
